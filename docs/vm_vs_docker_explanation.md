# Docker 사용 시 제약과 VM(권장) 사용 이유

## 요약

이 문서는 과제(SSH 포트 변경, 방화벽 활성화, 계정/그룹/ACL, cron, 로그 보존 등)를 Docker로 수행할 때 발생할 수 있는 제약과, 이를 해결하기 위해 VM(Ubuntu 22.04 등)을 사용해야 하는 이유를 정리합니다. 채점/검증 가능성과 재현성 관점에서 VM 사용을 권장합니다.

---

## 핵심 문제 요약

- 컨테이너는 호스트 커널을 공유하므로 시스템 데몬이나 커널 수준 네트워크 설정(UFW/firewalld, iptables/nftables 등)을 신뢰성 있게 제어하기 어렵다.
- 시스템 서비스 관리(`systemd`, `ssh`, `cron`, `logrotate`)가 컨테이너 환경에서 정상적으로 동작하지 않거나 특수 구성(예: privileged, host 네트워크)이 필요하다.
- 권한, ACL, 파일 시스템 소유자(uid/gid) 관련 증거가 컨테이너와 호스트 간에 달라질 수 있어 제출 증빙의 신뢰성이 떨어질 수 있다.

---

## 항목별 상세 제약

1. SSH 설정 및 포트 증명

- 문제: 컨테이너 내부에서 `sshd`를 띄우고 `sshd_config`를 수정할 수는 있으나, 외부 접속을 위해선 호스트에서 포트 포워딩(`-p`) 또는 host 네트워크 모드가 필요하다.
- 영향: 평가자가 "서버 수준"에서 포트 변경과 Root 차단을 검증하려면 호스트 기준의 설정 증거가 요구될 수 있음.

2. 방화벽(UFW/firewalld)

- 문제: 대부분의 컨테이너는 네임스페이스/권한 제약으로 `ufw enable` 또는 `firewall-cmd --state` 등의 명령이 실패하거나 호스트 규칙에 영향을 주지 못함.
- 영향: 내부에서 `ufw status`가 ‘활성화’로 보여도 실제 호스트 네트워크 정책과 분리되어 있어 평가자가 유효성을 의심할 수 있음.

3. systemd / 서비스 제어

- 문제: 컨테이너는 종종 systemd를 실행하지 않으며 `systemctl` 명령이 작동하지 않음.
- 영향: `sshd` 재시작, `cron` 데몬 관리, `logrotate` 예약 등 과제에서 기대하는 운영 흐름을 재현하기 어려움.

4. cron(주기 실행)

- 문제: cron 데몬이 컨테이너에 기본적으로 없거나, 별도 데몬을 띄워야 함. 단일 프로세스 패턴의 컨테이너에서는 cron이 죽거나 동작하지 않을 수 있음.
- 영향: “agent-admin의 crontab으로 매분 실행되어 monitor.log가 누적된다”는 요구 검증이 불안정해짐.

5. 권한, 소유자, ACL

- 문제: 이미지에서 생성한 계정(uid/gid)은 호스트와 매핑될 때 달라질 수 있음(특히 볼륨 마운트 시).
- 영향: `ls -l`, `getfacl`로 제출된 증거가 호스트 상태와 일치하지 않을 우려가 있어 채점자 신뢰도 저하.

6. 로그 보존 및 회전

- 문제: 로그가 컨테이너 파일시스템에 있으면 컨테이너 재시작/재배포 시 로그가 유실될 수 있음. `logrotate`는 데몬/스케줄러 의존적.
- 영향: 장기 보존/압축/아카이브 요구(예: 7일 이후 압축, 30일 이후 삭제) 재현 어려움.

7. 보안/심사 논리 왜곡

- 문제: 컨테이너 내부 root 권한과 호스트 root 권한의 차이로 인해 "Root 원격 접속 차단" 의도를 정확히 증명하기 어려움.

---

## 재현성 및 채점 리스크

- 동일한 Docker 이미지라도 호스트 네트워크/권한/옵션(`--privileged`, `--network host`, `-v /etc/ssh:/etc/ssh`)에 따라 결과가 크게 달라집니다. 채점 환경과 차이가 있으면 점수 누락 가능성이 큽니다.
- 과제의 의도(운영체제 레벨 설정·관리 능력 검증)를 충실히 증명하려면 호스트 수준에서 설정을 변경하고 확인할 수 있는 환경이 필요합니다.

---

## VM 사용(권장) 이유

- VM은 전체 OS 인스턴스를 분리된 커널과 사용자 공간으로 제공하므로 `sshd`, `ufw/firewalld`, `systemd`, `cron`, `logrotate` 등 시스템 서비스가 호스트와 동일한 방식으로 동작합니다.
- 계정(uid/gid), 파일 소유권, ACL, 포트 리스닝 상태, 방화벽 규칙 등 모든 증거를 OS 수준에서 일관되게 수집해 제출할 수 있습니다.
- 채점자가 요구하는 명령(`ss -tulnp`, `ufw status`, `firewall-cmd --list-all`, `getfacl`)의 출력이 호스트 레벨과 동일한 신뢰도를 가집니다.

---

## Docker로 꼭 수행해야 할 경우(대체 방안)

- 제출 전에 평가자에게 Docker 사용 허용 여부와 채점 기준을 반드시 확인한다.
- 가능한 경우 다음을 문서화해서 함께 제출한다:
  - 사용한 Docker 실행 옵션(예: `--privileged`, `--network host`, 볼륨 마운트 상세)
  - 컨테이너 내부에서 실행한 명령과 컨테이너 외부(호스트)에서 확인한 명령 출력 비교 증거
- 권장 구성(하지만 보안/신뢰성 저하를 수반함):
  - `--privileged` 또는 `--cap-add=NET_ADMIN --cap-add=SYS_ADMIN`
  - `--network host` (포트/방화벽 검증을 쉽게 하기 위함)
  - 호스트의 `/var/log`나 `/etc/ssh`를 볼륨으로 마운트하여 증거 일관성 확보

---

## 제출 시 권장 체크리스트(VM 기준)

- OS: Ubuntu 22.04 LTS (권장)
- SSH: `sshd_config` 변경(Port 20022, PermitRootLogin no) 및 `ss -tulnp` 출력 캡처
- 방화벽: UFW 또는 firewalld 활성화 및 허용 포트 설정 캡처
- 계정/그룹: `id agent-admin` 등 출력 캡처
- 권한/ACL: `ls -l`, `getfacl` 출력 캡처
- cron: `crontab -l -u agent-admin` 및 monitor.log의 시간별 누적 증거
- 로그: `/var/log/agent-app/monitor.log` 최근 라인 및 logrotate 설정(또는 스크립트)

---

## 결론

Docker는 경량화된 격리 환경으로 빠른 개발·테스트에는 유리하지만, 본 과제처럼 OS 레벨의 설정 변경과 운영 증거 수집이 필요한 과제에서는 증거 신뢰성과 채점 재현성 측면에서 VM 사용을 권장합니다. Docker를 선택할 경우 반드시 평가자 합의 및 추가 문서(호스트-컨테이너 매핑, 사용 옵션)를 준비해야 합니다.
